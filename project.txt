Daily AI Digest — Architecture & Approach
This is a beautifully scoped project, Vinit. It's simple enough to finish, but complex enough to genuinely demonstrate production MLOps/DevOps skills. Let me walk you through the full approach before we touch a single line of code.

The Big Picture
Think of this system as a pipeline factory with a subscription desk at the front.

Users walk up to the desk (React/FastAPI frontend) and leave their email.
Every morning, a factory worker (Celery beat scheduler) wakes up automatically.
The worker fetches raw news, hands it to an LLM chef (Gemini/OpenAI) who simplifies it.
The formatted digest gets emailed to every subscriber.
A quality inspector (MLflow) watches every LLM call and logs what happened.
Security cameras (Prometheus + Grafana) watch the whole factory 24/7.


Phase-by-Phase Breakdown
Phase 1 — Project Skeleton & Version Control
Goal: Clean repo structure from day one.

Initialize Git repo with a proper .gitignore (Python, Docker, env files).
Set up branch strategy: main (production), dev (active development), feature branches.
Structure the repo with clear separation: backend/, frontend/, infra/, ml/, tests/.
Set up pre-commit hooks (black, flake8, isort) so code stays clean automatically.


Phase 2 — Core Backend (FastAPI)
Goal: A working REST API with a database.
What you build:

POST /subscribe — accepts email, saves to PostgreSQL, sends a welcome email.
DELETE /unsubscribe — soft-deletes the user (marks inactive, never hard deletes).
GET /health — for uptime monitoring.
GET /digests — optional: let users view past digests.

Database schema (PostgreSQL):

subscribers table: id, email, is_active, subscribed_at.
digests table: id, date, content_json, sent_at.
digest_logs table: tracks which emails were sent to whom and whether they succeeded.
llm_logs table: prompt, response, model used, latency, token count — this feeds MLflow.

Tools: SQLAlchemy ORM + Alembic for migrations (just like your IoT platform).

Phase 3 — The News Fetching Pipeline
Goal: Reliable, fresh AI/ML content every day.
Sources (pick 2–3):

arXiv API — free, no key needed, returns latest ML papers. Filter by cs.AI, cs.LG, stat.ML categories.
NewsAPI — fetch articles from sources like TechCrunch, Wired, VentureBeat filtered by "artificial intelligence."
RSS feeds — Hugging Face blog, Google AI Blog, DeepMind blog. These are free and always fresh.

What the fetcher does:

Pulls 10–15 raw items.
Deduplicates (checks against digests table — don't summarize the same paper twice).
Scores by recency and picks the top 5.
Hands the list off to the LLM summarizer.


Phase 4 — LLM Summarization (The Core ML Part)
Goal: Turn dense research abstracts into 3-sentence plain English summaries.
The prompt strategy:
Each item gets a structured prompt asking the LLM to return:

A one-line headline.
A 2–3 sentence plain English summary (written for a curious non-expert).
A "Why it matters" section (1–2 sentences on real-world impact).
The original link.

You'll use Gemini 1.5 Flash (free tier is generous) or OpenAI GPT-4o-mini (cheap). Both work identically for this use case.
Output format: Structured JSON from the LLM — parse it, validate it, then render it into the email template.

Phase 5 — Celery + Redis (Background Jobs & Scheduling)
Goal: The digest runs automatically every morning without you doing anything.
Two types of Celery workers:

Celery Beat (scheduler): Acts like a cron job. At 7:00 AM every day, it fires a task called run_daily_digest.
Celery Worker (executor): Picks up that task and runs the full pipeline — fetch → summarize → format → email → log.

Why not just a cron job? Because Celery gives you retries on failure, task queuing, distributed execution, and visibility into what's running — things a plain cron job can't do.
Redis is just the message broker here — Celery puts tasks in Redis, workers pull from Redis.

Phase 6 — Email Sending
Goal: Beautiful, reliable emails that don't land in spam.

Use Resend (you already know it from your IoT platform — great choice).
Design a clean HTML email template with sections for each digest item.
Track send status per subscriber in digest_logs — if a send fails, mark it for retry.
Add an unsubscribe link at the bottom of every email (this is legally required in most countries).


Phase 7 — MLflow (LLMOps Tracking)
Goal: Treat every LLM call like an ML experiment you can analyze.
What you log for every digest generation run:

The prompt template version (so you can compare prompt v1 vs v2).
The model name and temperature used.
Token counts (input + output) — to track cost.
Latency (how long the LLM took).
The final output (so you can inspect quality).
A "quality score" — you can manually rate digests, or use another LLM call to auto-score readability.

Why this matters for your resume: You're not just calling an API. You're tracking LLM behavior systematically, which is exactly what LLMOps engineers do at companies like Cohere, Weights & Biases, and Databricks.

Phase 8 — Docker & Docker Compose
Goal: One command starts the entire system locally.
Services in docker-compose.yml:

api — FastAPI app.
worker — Celery worker.
beat — Celery beat scheduler.
db — PostgreSQL.
redis — Redis broker.
mlflow — MLflow tracking server (runs as its own service).
flower (optional) — Celery's web UI to monitor tasks visually.

Every service gets proper health checks and environment variable injection from a .env file.

Phase 9 — CI/CD with GitHub Actions
Goal: Every push to main automatically tests and deploys.
Two workflows:

ci.yml (runs on every PR):

Install dependencies.
Run pytest on your API routes and pipeline logic.
Run black --check and flake8 for code quality.
Build the Docker image to ensure it compiles.


cd.yml (runs on merge to main):

Build and push Docker images to Docker Hub or GitHub Container Registry.
SSH into your DigitalOcean droplet and run docker compose pull && docker compose up -d.



This means merging a PR automatically deploys to production — that's a real-world CI/CD pipeline.

Phase 10 — Monitoring (Prometheus + Grafana)
Goal: Know when something breaks before your users do.

Prometheus scrapes metrics from FastAPI (request count, latency, error rate) and Celery (task success/failure rate).
Grafana visualizes those metrics on a dashboard.
Set up alerts: if the daily digest task fails, send yourself an email or Slack notification.

This is the "lights on the factory floor" layer — everything is observable.

Optional: Kubernetes
If you want to go further, Kubernetes (K8s) lets you run this on a cluster instead of a single server. You'd convert your Docker Compose setup into Kubernetes manifests (Deployment, Service, CronJob for scheduling). This is advanced but extremely impressive on a resume. Only add this after everything else is stable.
